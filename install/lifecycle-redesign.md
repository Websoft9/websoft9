# 安装升级卸载简化重构方案

本文档基于 `main` 分支旧产品架构与实现事实，以及当前分支新产品架构与实现事实，给出下一版生命周期的简化设计方案。

目标不是做一个覆盖所有历史情况的复杂框架，而是基于行业最佳实践和 Websoft9 当前实际，定义一个简单、可实现、可验证的方案。

## 1. 设计目标

下一版生命周期只追求五件事：

1. 安装简单
2. 升级可分流
3. 失败可回退
4. 卸载可解释
5. 文档与代码统一

## 2. 基本原则

### 2.1 面向用户的原则

1. 单入口命令
2. 少量必要参数
3. 默认值清晰
4. 成功与失败都要有明确结果

### 2.2 面向工程的原则

1. 先识别环境，再决定路径
2. 先建立备份点，再做变更
3. 先冻结资产映射，再写迁移动作
4. 安装、迭代升级、跨代迁移、卸载四条路径不混写

### 2.3 结合 Docker 最佳实践后的简化原则

1. 持久数据优先落在命名卷，不依赖容器可写层
2. 卷生命周期与容器生命周期分离
3. 备份是升级前置动作，不是失败补救动作
4. 不混用 Docker 自身重启策略和宿主机 process manager 管理同一新运行时
5. 混合环境不默认自动处理

## 3. 下一版统一生命周期模型

下一版只保留四条明确路径：

1. 全新安装
2. 新版迭代升级
3. 旧版到新版迁移升级
4. 卸载

其中：

1. 全新安装只面向空环境
2. 迭代升级只面向明确的现代环境
3. 跨代迁移只面向明确的旧环境
4. `mixed` 和 `unknown` 默认不自动处理

## 4. 环境识别简化规则

### 4.1 识别输入信号

下一版环境识别只保留五类信号：

1. 容器名
2. 卷名
3. systemd 单元
4. 安装目录与 compose 文件，其中旧版还包括宿主机 `/data/compose` 应用堆栈目录与 `/data/websoft9/source` 安装源目录
5. 入口端口与配置文件

### 4.2 识别输出状态

识别结果只允许四种：

1. `empty`
2. `legacy`
3. `modern`
4. `mixed`

`unknown` 可以保留为内部调试状态，但不应成为用户态常规分支。

### 4.3 识别后的处理原则

1. `empty` 只允许安装
2. `modern` 只允许迭代升级或现代卸载
3. `legacy` 只允许跨代迁移或旧系统卸载
4. `mixed` 一律停止自动执行，转人工治理

### 4.4 当前代码事实下的识别锚点

为了避免后续实现时再次把历史兼容模型误写成当前运行时事实，第一版识别规则还应直接锚定到当前可核对的新旧代码实现：

1. 旧版本运行时锚点：`websoft9-apphub`、`websoft9-deployment`、`websoft9-git`、`websoft9-proxy` 四个容器，以及 `apphub_logs`、`apphub_config`、`portainer`、`gitea`、`nginx_data`、`nginx_letsencrypt`、`nginx_modsec`、`nginx_var` 等命名卷，外加宿主机 `/data/compose` 堆栈目录
2. 当前新版本运行时锚点：`websoft9` 单容器与 `websoft9_data` 主卷
3. 旧版本入口锚点：旧 `install.sh` 的 `console_port`、`app_http_gateway`、`app_https_gateway` 参数通过旧 `proxy` 容器统一暴露（对应 `.env` 中 `CONSOLE_PORT`、`APP_HTTP_GATEWAY`、`APP_HTTPS_GATEWAY`）
4. 当前新版本入口锚点：`CONSOLE_PORT` 暴露 `9000`，HTTP/HTTPS 当前在 compose 中固定暴露为 `80:80`、`443:443`

这意味着：

1. 第一版 `modern` 识别不应再把 `product_*` 卷当作当前新版本标准信号
2. 第一版入口参数设计不应先把 HTTP/HTTPS 端口重映射写成当前默认事实
3. 旧版命名卷只能作为识别信号，不能假设其内部目录结构与新版 `/data` 子树一一对应，详见第 7 章迁移设计
4. 如果未来运行时事实发生变化，应先更新本节锚点，再更新脚本和其余方案章节

## 5. 安装设计

### 5.1 安装定位

安装路径只负责把当前现代运行时部署到空环境。

它不负责：

1. 吸收旧系统
2. 兼容混合环境
3. 自动修复历史垃圾状态

### 5.2 安装最小步骤

安装只做下面步骤：

1. 识别环境必须为 `empty`
2. 检查 OS、磁盘、Docker、端口、路径写权限
3. 获取部署物料
4. 写入标准 `.env`
5. 启动单容器
6. 做健康检查
7. 记录安装结果

### 5.3 安装参数建议

如果严格对齐当前新版本真实运行时实现，第一版安装参数应先保守收敛。

当前新版本的 compose 事实是：

1. 控制台入口通过 `CONSOLE_PORT` 暴露
2. HTTP/HTTPS 当前仍以 `80:80`、`443:443` 固定暴露
3. 主数据卷当前是 `websoft9_data:/data`

因此，第一版建议先保留这些高价值参数：

1. `--channel`
2. `--version`
3. `--console-port`
4. `--path`
5. `--proxy`

`--http-port` 与 `--https-port` 可以保留为后续目标能力候选项，但在当前新版本运行时事实下，不应先写成第一版默认参数承诺。

不再把旧时代的 `--apps`、`--devto` 之类参数带入新标准。

### 5.4 安装动作清单与失败边界

全新安装在实现层建议只保留下面 4 个阶段：

1. 前置检查：OS、Docker、端口、路径、网络、磁盘
2. 物料准备：镜像、compose、`.env`、安装目录
3. 启动切换：启动单容器并等待健康状态
4. 成功确认：入口可达、产品 API 可达、核心运行时就绪

安装路径的失败边界也应保持简单：

1. 如果失败发生在“启动切换”前，直接退出，不需要回退模型
2. 如果失败发生在容器启动后，只需要清理本次安装生成的运行实体与临时物料
3. 安装路径默认不承担旧资产恢复责任，因为它只面向 `empty` 环境

### 5.5 全新安装实施表

| 维度 | 第一版要求 |
|---|---|
| 输入信号 | 环境识别结果为 `empty`；Docker 可用或可安装；控制台入口端口、路径、网络和磁盘满足前置条件 |
| 执行步骤 | 前置检查 -> 获取部署物料 -> 生成标准 `.env` -> 启动现代单容器 -> 写入安装结果 |
| 校验项 | 容器健康检查通过；`9000` 入口可达；`/api` 根路径可达；`websoft9_data` 已挂载到 `/data`；关键运行时进程完成启动 |
| 失败处理 | 启动前失败直接退出；启动后失败清理本次生成的容器、临时物料和未使用部署文件 |
| 非目标 | 不吸收旧系统；不处理 `legacy`、`mixed`、`unknown`；不自动修复历史垃圾状态 |

### 5.6 全新安装最小验收表

| 验收项 | 必须看到的结果 | 不通过时的判定 |
|---|---|---|
| 容器启动 | 现代容器成功启动并通过健康检查 | 视为安装失败 |
| 控制台入口 | `9000` 入口可达 | 视为安装失败 |
| API 基础可用 | `/api` 根路径或基础健康接口可达 | 视为安装失败 |
| 主数据根就绪 | `websoft9_data:/data` 已挂载且关键子树可读 | 视为安装失败 |
| 产品状态初始化 | 产品元数据、身份状态或基础配置初始化成功 | 视为安装失败 |

## 6. 新版迭代升级设计

### 6.1 定位

迭代升级是同模型升级，不应混入旧系统迁移逻辑。

### 6.2 最小步骤

1. 识别环境必须为 `modern`
2. 备份 `.env`、compose、镜像信息和主数据卷
3. 获取新部署物料与镜像
4. 做必要的数据结构迁移
5. 切换新容器
6. 做健康检查和产品级验证
7. 失败时回滚到升级前运行时

### 6.3 迭代升级的简化要求

1. 尽量不要求用户重新输入参数
2. 尽量复用已有卷与路径
3. 尽量避免新增命令模式
4. 只处理仓库中已经明确的现代数据模型

### 6.4 迭代升级动作清单与回退边界

迭代升级在实现层建议固定成下面 5 个阶段：

1. 升级前识别：确认 `modern`、记录当前版本、读取当前 `.env`
2. 升级前备份：备份 `.env`、compose、镜像标签、主数据卷和必要兼容卷
3. 物料切换：拉取新镜像、生成新部署物料、执行必要 schema 迁移
4. 启动验证：启动新容器，检查健康状态、入口和关键 API
5. 结果收口：成功则记录升级结果，失败则立即切回旧运行时

迭代升级的回退边界应明确为：

1. 新容器未通过健康检查，不进入数据清理阶段，直接恢复旧部署物料和旧镜像
2. 新容器健康但关键业务验证失败，允许回退到升级前卷快照或备份点
3. 只有在新运行时通过健康检查和关键业务验证后，才允许清理旧镜像或旧临时物料

### 6.5 新版迭代升级实施表

| 维度 | 第一版要求 |
|---|---|
| 输入信号 | 环境识别结果为 `modern`；当前 `.env`、compose、镜像标签和 `websoft9_data` 主卷可读取；不存在 `mixed` 信号 |
| 执行步骤 | 记录当前版本与参数 -> 备份配置、镜像和卷 -> 拉取新物料 -> 执行必要 schema 迁移 -> 启动新容器 -> 执行升级后验证 |
| 校验项 | 容器健康检查通过；入口可达；关键 API 可达；`/data` 关键目录结构可读；必要业务探针通过 |
| 回退动作 | 恢复升级前 `.env`、compose、镜像标签和 `websoft9_data` 备份点；重新启动旧运行时 |
| 禁止动作 | 在校验通过前删除旧镜像、旧部署物料或兼容卷；要求用户重新输入大量安装参数 |

### 6.6 新版迭代升级最小验收表

| 验收项 | 必须看到的结果 | 不通过时的判定 |
|---|---|---|
| 新运行时启动 | 新容器通过健康检查 | 视为升级失败，执行回退 |
| 入口连续性 | 原有控制台入口仍可达 | 视为升级失败，执行回退 |
| API 连续性 | 关键 API 可达且无明显初始化异常 | 视为升级失败，执行回退 |
| 主数据根连续性 | `/data` 关键目录结构可读，核心状态未丢失 | 视为升级失败或升级不完整 |
| 回退点有效 | 升级前配置、镜像和主卷备份点仍有效，直到收口完成 | 视为升级流程不合格 |

## 7. 旧版到新版迁移升级设计

### 7.1 定位

跨代迁移是独立路径，不应被包装成“普通 upgrade 的一个 if 分支”。

### 7.2 必须完成的动作

1. 识别环境必须为 `legacy`
2. 记录旧端口、旧路径、旧 compose、旧 `.env`、旧 systemd 状态
3. 备份所有必需旧卷和关键可选卷
4. 冻结资产映射计划
5. 准备新部署物料
6. 停止旧运行时
7. 导入新数据根
8. 启动新容器
9. 做入口、数据、服务级验证
10. 失败时恢复旧运行时

### 7.3 跨代迁移的核心原则

1. 全自动、零人工：从识别旧环境到完成新版接管的全过程不需要任何人工录入或人工搬运，配置与数据尤其必须自动迁移
2. 配置与数据双自动：旧版第三方凭据、产品版本、应用编排、仓库内容、域名代理与证书绑定全部自动接续，用户迁移后用**旧凭据**即可登录，无需重置
3. 只覆盖主路径旧系统，不强行兼容所有未知历史变种；落不进迁移映射的未知资产转人工治理，但主清单内资产必须全自动
4. 旧系统稳定回退能力优先：在新版通过验收前，旧运行时与旧卷必须保持可回退

### 7.4 建议冻结的第一版迁移资产

> 关键前提（代码已核实）：新版平台**本就是为吸收既有第三方数据并自动对账而设计的**，因此跨代迁移可以做到完全自动、零人工，配置与数据一并自动接续。核心机制有四：
>
> 1. 凭据来自文件、可被预置：新版 `IntegrationCredentialProvider` 优先从 `/data/gitea/credential`、`/data/portainer/credential`、`/data/nginx-proxy-manager/credential.json` 读取第三方凭据（缺失时再回退 apphub config.ini）。迁移阶段从旧 `config.ini` 抽取旧密码合成这些凭据文件，新平台启动后即用**旧密码**与迁移后的服务对账，无需任何人工录入。
> 2. NPM 用户库自动对账：`sync_npm_credentials` 会直接改写 NPM sqlite（`/data/database.sqlite`）的用户名/邮箱并重算 bcrypt 密码以匹配凭据文件；新版 NPM 数据库路径 `/data/database.sqlite` 与旧 `nginx_data` 卷根的 `database.sqlite` 一致，jc21 2.12.6 启动时自动跑 knex 迁移升级库结构。旧 NPM 数据按文件选择性搬入（`database.sqlite`+`nginx/`，排除会与新版 `/data/logs` 冲突的旧 `logs/`）后即被自动接管。
> 3. Gitea / Portainer 启动自迁移：迁移时保留旧 Gitea 的 `SECRET_KEY`/`INTERNAL_TOKEN` 与数据库，并把仓库/app data 路径重映射到新布局（旧 `git/repositories`→新 `/data/gitea/git/repositories`），端口/ROOT_URL 由新 start 脚本的 `reconcile_config` 启动时自动改写为新值（3001、`/w9git/`），加密字段继续有效；Portainer 启动时自动向前迁移其 bolt 数据库，且新容器挂载 `/var/run/docker.sock`，已部署应用容器继续运行。两者凭据文件被新平台 honor，认证状态延续。
> 4. 产品版本自动推断：`migrate_product_runtime_state(legacy_system_ini_file=...)` 读取旧 `system.ini` 的 `[max_apps] key` 自动推断版本（2=free、3=starter、10=standard、缺省或 ≥10000=enterprise）并写入新版运行时状态。
>
> 因此迁移的本质是"卷资产确定性转换搬入新 `/data` 子树 + 从旧配置自动合成凭据文件 + 依赖新版内置对账完成接管"，三者全部自动，配置与数据都不需要人工参与。

第一版自动迁移主清单（全部全自动、零人工，含配置与数据）：

1. 旧版应用部署定义（宿主机 bind `/data/compose` 迁入命名卷 + Portainer stack）
2. 自定义 Compose 源与仓库内容（Gitea 仓库 + 加密密钥）
3. 域名、代理与访问绑定（NPM 数据库）及其证书
4. 第三方登录凭据（从旧 config.ini 合成凭据文件，延续旧密码）
5. 产品版本与运行时状态（从旧 system.ini 自动推断）

下表给出每类资产的旧位置、新位置与自动处理方式（均已对代码核实）：

| 资产 | 旧位置 | 新位置 | 自动处理方式 |
|---|---|---|---|
| NPM 数据库与代理配置 | `nginx_data` 卷（卷根含 `database.sqlite` + `nginx/` + `custom_ssl/` + `letsencrypt/` + `logs/`） | `/data/database.sqlite` + `/data/nginx/*` | **按文件选择性搬入**：只搬 `database.sqlite` 与 `nginx/`，**不整卷根 dump**（旧卷根的 `logs/` 会与新版 `/data/logs` 冲突，必须排除）；NPM 2.12.6 启动跑 knex 迁移自升级；`sync_npm_credentials` 用合成凭据对账用户名/密码 |
| 自定义与签发证书 | `nginx_data` 卷 `custom_ssl/` + `nginx_letsencrypt` 卷 | `/data/nginx-proxy-manager/custom_ssl` + 新证书目录 | 搬入并自动接管，无需人工重签 |
| Gitea 仓库与加密状态 | `gitea` 卷（卷根 `gitea/` + `git/` 布局：仓库在 `git/repositories`、app data 在 `gitea/`、app.ini 在 `gitea/conf`、端口 3000） | `/data/gitea`（`conf/app.ini`、`data/`、`git/repositories`、`gitea.db`、端口 3001） | 结构转换搬入并**重映射路径**：旧 `git/repositories`→新 `/data/gitea/git/repositories`、旧 app data→新 `/data/gitea/data`、旧 `gitea.db`→新 `/data/gitea/gitea.db`；保留旧 `app.ini` 的 `SECRET_KEY`/`INTERNAL_TOKEN`，端口/ROOT_URL 由新 start 脚本的 `reconcile_config` 启动时自动改写为新值（3001、`/w9git/`）；检测到已存在配置即不再重生成密钥 |
| Portainer 编排状态 | `portainer` 卷 | `/data/portainer` | 搬入；Portainer 启动自动迁移其 bolt 数据库，stack 继续可解释；新容器挂载 `/var/run/docker.sock`，已部署应用容器继续运行、可被新 Portainer 接管 |
| 应用 Compose 源 | **宿主机 bind** `/data/compose`（旧 deployment 容器挂载 `/data/compose:/data/compose`） | `websoft9_data` 命名卷内的 `/data/compose`（新 compose 仅挂 `websoft9_data:/data`，无 `/data/compose` 宿主机 bind） | 路径字符串 `/data/compose` 不变（Portainer 引用一致），但**后端存储从宿主机目录改为命名卷**，必须把宿主机 `/data/compose/*` 搬进命名卷子树 |
| 第三方登录凭据 | 旧 `apphub_config` 卷 `config.ini` 的 `[gitea]`/`[portainer]`/`[nginx_proxy_manager]` 段 | `/data/gitea/credential`、`/data/portainer/credential`、`/data/nginx-proxy-manager/credential.json` | 抽取旧用户名/密码合成凭据文件，新平台用旧密码对账，登录无需重置 |
| API key | 旧 `config.ini` 的 `[api_key]` | 新 apphub 配置 api_key | 自动写入新配置模型 |
| 产品版本 | 旧 `system.ini` 的 `[max_apps] key` | 新版 `product_runtime_state` | `migrate_product_runtime_state` 自动推断 edition 并写入 |
| AppHub 历史日志 | `apphub_logs` 卷 | `/data/logs/legacy-apphub` | 自动归档保留（非阻塞资产，不影响接管） |

> 说明：旧 `apphub_config` 的 `config.ini` 本身不整文件导入（其中旧多容器 DNS `websoft9-proxy:81`、`websoft9-git:3000`、`websoft9-deployment:9000` 已失效），但其中**有意义的字段（第三方凭据、api_key、版本来源）会被自动抽取并写入新版对应的凭据文件、配置模型与运行时状态**——所以"配置迁移"同样是全自动的，而不是人工转写。

迁移边界：

1. 主清单内资产全部全自动迁移（含配置与数据），不需要任何人工录入或人工搬运
2. ModSecurity 配置、旧 Cockpit 插件目录、宿主机级定制脚本等旧壳时代资产不属于新版运行时所需，默认不迁移；其中 ModSecurity 旧数据随卷备份保留以便回退
3. 落不进上面映射表的未知历史资产转人工治理，但不得因此降低主清单的自动化程度

### 7.5 迁移设计输入来源

跨代迁移设计应直接使用新旧能力矩阵中的“迁移关注点”列作为输入，并区分三类对象：

1. 原生产品能力：产品配置、身份、版本、日志与内部状态。新版身份/网关密钥虽由新版生成，但**旧版本（system.ini）与旧 api_key 等可被自动抽取接续**
2. 集成封装能力：第三方系统上的编排状态、仓库内容、访问配置与代理关系，全部通过“卷转换 + 凭据合成 + 内置对账”自动接续
3. 第三方入口能力：入口替换与会话桥接问题，做入口替换即可，不需要复杂数据迁移

判定纪律：被归为“集成封装能力”的对象，默认要把第三方状态（卷数据）与产品侧凭据（凭据文件）成对自动处理；被归为“第三方入口能力”的对象，默认只做入口替换。

### 7.5A 能力分类与自动迁移处理对照表

| 能力分类 | 典型对象 | 自动迁移处理方式 | 设计含义 |
|---|---|---|---|
| 原生产品能力 | 产品身份、产品配置、版本、运行时日志、`/data` 内部状态 | 身份/网关密钥由新版自生成；版本从旧 `system.ini` 自动推断；api_key 等业务字段自动抽取；历史日志自动归档 | 旧 config.ini 不整体导入，但其中有意义字段被自动接续，无需人工转写 |
| 集成封装能力 | App Store、My Apps、Compose Apps、App Access、代理业务数据 | 卷确定性转换搬入新 `/data` 子树 + 从旧 config.ini 合成凭据文件 + 依赖 `sync_npm_credentials`/gitea-portainer 凭据 honor/各组件 DB 自迁移 | 跨代切换后关键业务可继续使用，且用旧凭据即可登录，全程零人工 |
| 第三方入口能力 | Cockpit plugin 页面壳、第三方产品入口、宿主机桥接入口 | 做入口替换；不纳入数据迁移 | 入口承载方式问题，新版用 Terminal/Host Access 等替代 |

默认纪律：

1. 原生产品能力：优先自动抽取旧版可接续字段（版本、api_key），其余由新版自生成
2. 集成封装能力：卷数据与凭据成对自动处理，依赖新版内置对账，禁止把迁移降级为人工录入
3. 第三方入口能力：只做入口替换与历史保留

### 7.5B 统一迁移决策索引表

| 能力带 | 旧版来源 | 新版承接 | 自动迁移策略（零人工） |
|---|---|---|---|
| 产品身份与配置 | 旧 `apphub_config` 的 config.ini | 新版自生成身份/网关密钥；api_key 等字段自动抽取 | 不整体导入旧配置；自动抽取可接续字段写入新模型，身份由新版生成 |
| 产品版本 | 旧 `system.ini` 的 `[max_apps]` | `product_runtime_state` | `migrate_product_runtime_state` 自动推断 edition 写入 |
| 日志与运行时状态 | 旧 apphub 日志/宿主机痕迹 | 新版 `/data/logs` | 自动归档到 `/data/logs/legacy-apphub`，不阻塞接管 |
| 应用编排与生命周期 | Portainer + 宿主机 bind `/data/compose` | Portainer + 命名卷内 `/data/compose` | 搬入 `portainer` 卷与宿主机 `/data/compose/*`（后者迁入命名卷），Portainer 启动自迁移 bolt db，stack 继续可解释 |
| 仓库与 Compose 源内容 | `gitea` 卷（仓库在 `git/repositories`、端口 3000） | `/data/gitea`（`git/repositories`、端口 3001） | 结构转换搬入并重映射路径，保留 `SECRET_KEY`/db，端口/ROOT_URL 由 `reconcile_config` 启动自动改写；加密字段继续有效 |
| 域名、代理与访问绑定 | `nginx_data` 卷（NPM db） | `/data/database.sqlite` + `/data/nginx/*` | 按文件选择性搬入（只 `database.sqlite`+`nginx/`，排除会与 `/data/logs` 冲突的旧 `logs/`）；NPM 启动自迁移 schema；`sync_npm_credentials` 对账用户 |
| 第三方登录凭据 | 旧 config.ini 三个集成段 | 三个凭据文件 | 抽取旧密码合成凭据文件，新平台用旧密码对账，登录无需重置 |
| 证书 | `nginx_data` custom_ssl + `nginx_letsencrypt` | NPM 证书目录 | 搬入并自动接管 |
| 第三方页面入口 | Cockpit plugin/页面壳 | 第三方入口能力 | 做入口替换，不纳入数据迁移 |
| 宿主机桥接能力 | Cockpit 宿主机桥接 | Terminal / Host Access | 不做宿主机级自动继承，保留回退条件 |

落不进本表任一行的对象，说明尚未被清晰归类，应转人工治理，但不得降低主清单资产的自动化程度。

### 7.6 跨代迁移动作清单与回退边界

跨代迁移在实现层固定为下面 8 个阶段，全部自动执行：

1. 识别：确认 `legacy`，记录旧入口、旧卷、旧 `/data/compose`、旧 systemd、旧安装目录、旧 config.ini 与 system.ini
2. 备份：对旧卷、旧 config.ini、旧证书、旧 `/data/compose` 与旧控制面状态生成回退点
3. 停旧运行时：`docker compose -p websoft9 down` 停掉旧 4 容器并 `systemctl stop` 旧 `websoft9`/`cockpit.socket`/`cockpit`，**释放 80/443/9000 端口并停止旧卷写入**，确保新容器可抢占入口、卷转换时数据一致（此阶段只停用、不删除，便于回退）
4. 卷转换：把旧卷确定性转换搬入新 `/data` 子树——NPM 按文件选择性搬入（`database.sqlite`+`nginx/`，排除旧 `logs/`）、gitea 重映射路径（`git/repositories`→`/data/gitea/git/repositories`）并保留密钥、portainer 子树整搬、证书搬入、宿主机 `/data/compose/*` 迁入命名卷
5. 凭据合成：从旧 config.ini 抽取第三方密码生成三个凭据文件，从 system.ini 写入产品版本，从 `[api_key]` 写入新配置
6. 启动接管：启动新容器，平台内置对账完成（NPM 用户改写、gitea/portainer 凭据 honor、各 DB 自迁移、版本推断）
7. 验证：执行入口、数据、服务、凭据、版本级自动验证
8. 收口：通过则保留旧系统回退窗口待关闭，失败则自动恢复旧运行时

回退边界（仅宿主机级清理保持保守，数据迁移本身全自动）：

1. 新版通过验收前，旧 Cockpit、旧 systemd、旧卷与旧 `/data/compose` 备份不得不可逆删除（停旧运行时阶段只 `stop`/`down`，不 `disable`/不删除单元与镜像）
2. 失败发生在“启动接管”前，直接恢复旧运行时（重启旧容器与旧 systemd），不留半切换状态
3. 失败发生在“启动接管”后，优先恢复旧入口与旧运行时，再处理新系统残留
4. 旧 Cockpit、旧插件目录与宿主机级改动默认不在迁移窗口内自动清理，由独立卸载开关在回退窗口关闭后处理

### 7.7 旧版到新版迁移升级实施表

| 维度 | 第一版要求 |
|---|---|
| 输入信号 | 环境识别为 `legacy`；旧容器、旧卷、旧 systemd、旧入口、旧安装目录、旧 config.ini/system.ini 信号可观察；不存在 `mixed` |
| 执行步骤 | 记录旧环境 -> 生成回退点 -> 停旧运行时与旧 systemd 释放 80/443/9000 -> 卷确定性转换搬入新 `/data` 子树 -> 从旧 config.ini/system.ini 合成凭据文件/版本/配置 -> 启动新容器触发内置对账 -> 切换入口 -> 执行关键验证 |
| 校验项 | 新入口可达；新 `/data` 子树关键数据已就位；NPM/Gitea/Portainer 凭据对账成功且可用旧密码登录；编排/仓库/代理状态可解释；产品版本已从旧 system.ini 推断 |
| 回退动作 | 恢复旧入口；恢复旧 compose、旧 `.env`、旧卷与旧控制面；重启旧 Cockpit、多容器运行时与 systemd |
| 自动化边界 | 主清单资产（含配置与数据）全自动迁移，零人工；仅宿主机级旧壳清理延后到回退窗口关闭；`mixed`/`unknown` 转人工治理 |

### 7.8 旧版到新版迁移最小验收表

为了避免后续把“容器起来了”误当成“迁移完成”，旧版到新版迁移至少应通过下面这张最小验收表：

| 验收项 | 必须看到的结果 | 不通过时的判定 |
|---|---|---|
| 新入口切换 | `9000` 入口可达，旧入口切换结果可解释 | 视为迁移失败，优先恢复旧入口 |
| API 基础可用 | `/api` 根路径或基础健康接口可达 | 视为迁移失败，不能进入收口阶段 |
| 主数据根就绪 | `websoft9_data:/data` 已挂载，`/data/config`、`/data/logs`、`/data/gitea`、`/data/portainer` 等关键子树可读 | 视为迁移失败或迁移不完整 |
| 身份与产品状态 | 产品元数据、身份状态、关键配置可读取 | 视为迁移失败，不能宣称产品已接管 |
| 编排状态可解释 | Portainer / Gitea / 代理相关关键状态可解释，不要求一次性证明所有深层业务都完全正常 | 视为迁移不完整，不能关闭回退窗口 |
| 旧凭据延续可登录 | 迁移后可用**旧** Gitea / Portainer / NPM / 产品登录凭据直接登录新平台（凭据文件已从旧 config.ini 自动合成并对账成功） | 视为迁移不完整，说明凭据自动接续失败 |
| 配置与版本已接续 | 产品版本已从旧 system.ini 自动推断，api_key 等关键业务字段已自动写入新配置 | 视为迁移不完整，配置未自动接续 |
| 回退点仍有效 | 旧 compose、旧 `.env`、旧卷和旧控制面恢复条件仍在 | 视为迁移流程不合格，即使新容器已启动也不能收口 |

第一版还应明确两条验收纪律：

1. 只要上表中任何一项失败，就不能把本次动作标记为“迁移成功”
2. 配置与数据迁移（含证书、凭据、版本）必须在最小成功判定内完成；仅旧 Cockpit 清理、旧插件目录与旧宿主机脚本收尾属于后置收尾项，由回退窗口关闭后的独立开关处理，不并入最小成功判定

## 8. 卸载设计

### 8.1 统一卸载模式

下一版只保留三种卸载模式：

1. 停用
2. 标准卸载
3. 彻底清理

### 8.2 卸载分流规则

1. `modern` 使用现代卸载逻辑
2. `legacy` 使用旧系统卸载逻辑
3. `mixed` 不自动卸载

### 8.3 卸载的简化原则

1. 默认不删数据
2. 删除数据必须显式选择
3. 删除旧 Cockpit/systemd 遗留必须作为独立开关
4. 用户必须知道删掉了什么，保留了什么

### 8.4 卸载动作清单

无论是现代卸载还是旧系统卸载，动作层都建议固定回答 4 个问题：

1. 要停止哪些运行实体
2. 要删除哪些部署物料
3. 要保留还是删除哪些数据和镜像
4. 要不要处理旧 Cockpit、systemd 和宿主机遗留

只有这 4 个问题都能给出明确结果，卸载路径才适合自动化实现。

### 8.5 卸载实施表

| 维度 | 第一版要求 |
|---|---|
| 输入信号 | 环境识别结果为 `modern` 或 `legacy`；目标卸载模式明确为停用、标准卸载或彻底清理 |
| 执行步骤 | 识别环境与模式 -> 停止运行实体 -> 删除或保留部署物料 -> 删除或保留数据与镜像 -> 输出结果清单 |
| 校验项 | 目标容器或服务已停止；预期保留的数据仍存在；预期删除的部署物料已清理；结果清单可解释 |
| 安全边界 | `mixed` 不自动卸载；删除数据必须显式选择；删除旧 Cockpit、旧 systemd 和旧证书必须作为独立确认项 |
| 结果要求 | 用户必须明确知道删掉了什么、保留了什么、哪些旧控制面遗留尚未处理 |

### 8.6 卸载最小验收表

| 验收项 | 必须看到的结果 | 不通过时的判定 |
|---|---|---|
| 运行实体处理 | 目标容器或服务已按模式停止或删除 | 视为卸载失败 |
| 数据处理结果 | 用户选择保留的数据仍在；选择删除的数据已按预期清理 | 视为卸载失败或卸载结果不可信 |
| 部署物料处理 | 目标部署物料已按模式删除或保留 | 视为卸载失败 |
| 结果清单 | 用户可明确看到删除项、保留项和未处理遗留项 | 视为卸载流程不合格 |

## 9. 下一版脚本结构建议

如果后续重写脚本，建议结构上不要继续把所有逻辑堆在一个超长文件里。

建议拆成下面几层：

1. `install.sh` 只做参数入口和流程分发
2. `lib/detect.sh` 负责环境识别
3. `lib/install.sh` 负责全新安装
4. `lib/upgrade-modern.sh` 负责迭代升级
5. `lib/upgrade-legacy.sh` 负责跨代迁移
6. `lib/uninstall.sh` 负责卸载
7. `lib/backup.sh` 负责备份与回退

如果不想拆文件过多，最低限度也应拆成：

1. 入口层
2. 检测层
3. 路径实现层
4. 备份回退层

### 9.1 模块职责矩阵

为了避免后续脚本重写时再次把多条路径混写，模块职责建议固定如下：

| 模块 | 只负责什么 | 明确不负责什么 |
|---|---|---|
| `install.sh` | 参数解析、模式分发、统一日志入口、统一退出码 | 不直接写迁移细节、卷复制细节、环境识别细节 |
| `lib/detect.sh` | 读取容器、卷、systemd、路径、端口和配置文件信号；输出 `empty`、`legacy`、`modern`、`mixed` | 不直接做安装、升级、卸载动作 |
| `lib/install.sh` | 全新安装路径 | 不处理旧系统迁移，不处理混合环境治理 |
| `lib/upgrade-modern.sh` | 新版迭代升级路径 | 不处理旧卷映射，不处理 Cockpit 退场 |
| `lib/upgrade-legacy.sh` | 旧版到新版迁移路径 | 不承接 `modern` 环境，不处理未知资产自动迁移 |
| `lib/uninstall.sh` | 现代卸载与旧系统卸载分流 | 不隐式删除数据，不隐式清理旧控制面遗留 |
| `lib/backup.sh` | 配置、卷、镜像、部署物料和入口相关备份与恢复 | 不决定业务路径，不直接决定升级策略 |
| `lib/validate.sh` | 健康检查、入口验证、关键 API 验证、结果收口 | 不做数据迁移和环境识别 |

### 9.2 推荐命令面

下一版入口命令建议保持单入口，但在内部固定成有限模式：

| 模式 | 目标环境 | 作用 |
|---|---|---|
| `install` | `empty` | 全新安装现代运行时 |
| `upgrade` | 自动分流 `modern` 或 `legacy` | 用户视角单入口升级，内部按识别结果转迭代升级或跨代迁移 |
| `uninstall` | `modern` 或 `legacy` | 按环境分流卸载逻辑 |
| `detect` | 任意 | 输出识别结果和关键观察信号，用于人工审计与脚本调试 |
| `backup` | `modern` 或 `legacy` | 显式生成备份点，不与升级动作强耦合 |

第一版不建议暴露过多模式。尤其不建议把 `upgrade-modern`、`upgrade-legacy` 暴露成普通用户常用命令，它们更适合作为内部路径名称。

### 9.3 推荐参数面

为了保持用户入口简单，参数面建议分成三层：

1. 通用参数：`--channel`、`--version`、`--path`、`--proxy`
2. 当前事实对齐的入口参数：`--console-port`
3. 安全参数：`--mode`、`--force`、`--dry-run`、`--yes`、`--keep-data`

约束建议如下：

1. `--dry-run` 至少要支持 `detect`、`upgrade`、`uninstall` 的前置检查阶段
2. `--force` 不能绕过 `mixed` 环境停止规则，只能放宽非破坏性检查
3. `--keep-data` 只影响卸载与失败清理，不影响升级前强制备份
4. `--http-port` 与 `--https-port` 只有在未来部署物料显式支持网关端口重映射后才进入第一版正式参数面
5. 旧时代遗留参数不进入第一版统一命令面，除非后续证明仍有代码事实依赖

## 10. 实施前门槛

在开始重写脚本前，必须先确认以下问题都有明确答案：

1. 现代主卷标准固定为 `websoft9_data`，历史卷级拆分资产在迁移中作为被转换的输入源，转换搬入新 `/data` 子树后即对齐现代模型
2. 旧证书在迁移中自动搬入并接管（custom_ssl + letsencrypt），不是仅归档
3. 旧 Cockpit 包和插件目录在迁移成功后默认保留停用，由独立开关在回退窗口关闭后清理
4. 现代运行时成功判定要同时看容器健康与关键业务接口（入口 + API + 凭据对账 + 版本接续）
5. 旧版 Gitea/Portainer/NPM 数据通过“卷确定性转换 + 各组件启动自迁移 + 凭据文件对账”全自动接管；转换需对齐新版路径并保留 gitea 密钥与各 DB，禁止的是“搬错路径的裸拷贝”，不是禁止自动迁移
6. 旧第三方密码从旧 config.ini 抽取合成新凭据文件，新平台用旧密码对账，用户迁移后用旧凭据登录，无需人工交付新凭据
7. 旧 apphub config.ini 不整体导入，但其中第三方凭据、api_key、版本来源会被自动抽取写入新版凭据文件/配置/运行时状态——配置迁移同样全自动

### 10.1 第一版推荐默认决策

为了避免脚本重写时又回到开放式讨论，第一版建议先冻结下面 4 个默认决策：

| 决策项 | 第一版推荐默认值 | 理由 |
|---|---|---|
| 现代数据卷命名 | 继续以 `websoft9_data` 为现代主卷标准，所有持久化状态落在 `/data`；旧卷在迁移中作为被转换的输入源 | 与当前 compose 和入口脚本一致，迁移即把旧卷转换搬入新 `/data` 子树 |
| 旧证书策略 | 迁移时自动搬入 custom_ssl 与 letsencrypt 并由新版 NPM 接管，无需人工重签 | 新版 NPM 数据库与证书目录布局可直接承接旧证书，自动迁移最符合“零人工”目标 |
| 旧 Cockpit / 插件目录策略 | 迁移成功后默认保留并停用，是否清理由独立开关在回退窗口关闭后控制 | 跨代升级首先是控制面替换，不在同一窗口叠加不可逆宿主机清理 |
| 旧版第三方数据迁移方式 | 默认对 `gitea`/`portainer`/`nginx_data` 卷做确定性结构转换搬入新 `/data` 子树（对齐新路径、保留 gitea 密钥与各 DB），叠加各组件启动自迁移与凭据文件对账，全程零人工 | 新版内置 `sync_npm_credentials`、gitea/portainer 凭据 honor、各组件 DB 自迁移，专为吸收既有第三方数据设计；禁止的是搬错路径的裸拷贝，而非自动迁移本身 |
| 旧版凭据迁移方式 | 从旧 config.ini 抽取第三方密码合成 `/data/gitea/credential` 等凭据文件，新平台用**旧密码**对账，用户迁移后用旧凭据登录 | `IntegrationCredentialProvider` 优先读凭据文件，预置即可让认证状态延续，无需人工交付或重置 |
| 旧版 apphub 配置策略 | 不整体导入 config.ini，但自动抽取其中第三方凭据、api_key 与版本来源写入新版凭据文件/配置/运行时状态 | 旧 config.ini 多容器 DNS 已失效不能整体导入，但有意义字段可被自动接续，使配置迁移同样全自动 |
| 现代运行时成功判定 | 必须同时满足“容器健康 + 入口可达 + 关键 API 可达 + 凭据对账成功 + 版本接续 + 关键产品状态初始化”，不允许只看容器健康 | 单看容器健康不足以证明迁移后的产品壳、API、凭据与状态已就绪 |

如果后续需要改变这些默认值，应先修改本表，再允许脚本实现跟进；不应在脚本里先实现再倒推文档。

### 10.2 第一版非目标决策

为了继续保持第一版简单化，下面这些动作明确不作为默认目标：

1. 不在第一版里重新引入新的现代卷级拆分模型；现代目标模型继续以 `websoft9_data:/data` 为准
2. 不把“容器在跑”误写成“升级成功”
3. 不把旧 Cockpit 包、插件目录和宿主机脚本在迁移窗口内自动彻底清理写成默认行为（由回退窗口关闭后的独立开关处理）
4. 不把旧 config.ini 整文件导入新版（仅自动抽取其中可接续字段）
5. 不把“搬错路径的裸卷拷贝”当作迁移手段（迁移采用对齐新路径的确定性结构转换）
6. 不把旧版 apphub 配置自动导入新版当作默认迁移手段

## 11. 统一校验标准

四条路径在动作不同，但成功判定应尽量共享同一套审查口径。

### 11.1 校验层级

第一版统一只看 4 层校验：

1. 运行时校验：容器或服务是否完成启动，健康检查是否通过
2. 入口校验：控制台入口、HTTP/HTTPS 入口、API 根路径是否可达
3. 产品状态校验：产品元数据、身份状态、关键配置是否已初始化或可读取
4. 能力边界校验：只检查第一版承诺的最小能力，不扩展到所有页面和所有第三方功能

### 11.2 按路径的成功判定表

| 路径 | 必须通过的校验 | 可以延后处理的项 | 不作为第一版阻塞项 |
|---|---|---|---|
| 全新安装 | 容器健康通过；`9000` 入口可达；`/api` 可达；产品元数据初始化成功 | 文档里未承诺的第三方集成页面细节 | 旧系统兼容资产，因为 `install` 只面向 `empty` |
| 新版迭代升级 | 容器健康通过；原有入口仍可达；关键 API 可达；主数据卷结构和产品状态可读取 | 已归档兼容资产的整理 | 非关键页面样式或非核心集成细节 |
| 旧版到新版迁移升级 | 新入口可达；关键迁移资产存在；产品身份和配置可读取；回退点仍有效 | Cockpit 清理、旧证书接管、旧插件目录收尾 | 未冻结资产和未知历史变种 |
| 卸载 | 目标运行实体已停止或删除；用户选择保留的数据仍在；结果清单完整可解释 | 宿主机级遗留的后续人工清理 | `mixed` 环境自动治理 |

### 11.3 第一版关键校验项清单

无论是哪条路径，下面项目只要有一项失败，就不能宣称本次动作成功：

1. 现代容器未通过健康检查
2. `9000` 产品入口不可达
3. `/api` 根路径不可达或关键 API 不可用
4. 产品元数据或身份初始化失败
5. 对应路径要求保留的回退点未生成或已失效

### 11.4 第一版校验边界

为了避免把第一版做成无限责任模型，校验边界也要明确：

1. 不要求一次动作覆盖所有控制台页面的逐页验收
2. 不要求自动证明所有第三方集成系统的深层业务都完全正常
3. 对集成封装能力，第一版只要求“状态可解释、入口可达、关键元数据存在”
4. 对第三方入口能力，第一版只要求“入口替换正确、回退条件保留、数据未被误删”

## 12. Mixed 与人工治理边界

当前方案已经明确 `mixed` 和 `unknown` 不自动处理，但还需要把“人工治理”说得更具体。

### 12.1 进入人工治理的触发条件

出现下面任一情况，都应退出自动路径并转人工治理：

1. 现代容器和旧容器、旧卷同时存在
2. 旧入口和新入口端口冲突且无法通过标准映射解释
3. 关键资产来源不清，无法放入正式资产映射表
4. 关键回退条件无法建立，例如旧卷无法备份或旧 compose 缺失且容器状态不可恢复

### 12.2 人工治理后的可回归条件

人工治理的目标不是直接完成升级，而是把环境收敛回可自动处理状态。

允许重新进入自动路径前，至少要满足：

1. 环境识别结果重新稳定为 `empty`、`modern` 或 `legacy` 之一
2. 关键资产已能映射到正式资产表
3. 回退点已能建立
4. 当前目标路径的输入信号不再互相冲突

## 13. 当前建议结论

当前最合理的推进方式不是继续修补现有 `install.sh`，而是：

1. 先以本目录文档冻结代码事实
2. 再统一现代数据模型和迁移边界
3. 最后用简化后的生命周期模型重写脚本

这会比继续在现有脚本上局部打补丁更稳，也更符合项目当前架构已经整体切换的现实。
