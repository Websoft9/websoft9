# 平台生命周期与升级迁移规范

创建时间：2026-06-08
状态：执行草案
适用对象：DevOps、发布负责人、安装脚本维护者、平台运维
所属阶段：Phase 5 平台生命周期治理执行包

## 1. 文档定位

这份文档负责平台本身的安装、升级、卸载和兼容问题，不负责应用商店数据制品治理。

本目录的阅读入口统一以 [README.md](./README.md) 为准。

如果需要理解与发布治理的协同边界，可参考 [../standards/devops/roadmap_cn.md](../standards/devops/roadmap_cn.md)。

它统一回答四类问题：

1. 当前平台安装与卸载入口到底是什么
2. 平台升级链路今天有什么缺口
3. 旧模型与新模型之间的兼容和迁移边界是什么
4. 这部分后续应该按什么顺序落地

## 2. 主干旧模式与当前事实

以下内容以当前仓库可验证事实为基线。

### 2.1 当前安装与卸载入口

根据 [install/README.md](../../install/README.md)：

1. 当前对外安装入口是 `install/install.sh`
2. 当前对外卸载入口是 `install/uninstall.sh`
3. 当前支持独立 Docker 安装入口 `install/install_docker.sh`
4. 当前安装模型已经面向单容器运行时，而不是旧的 Cockpit 多容器部署模式

### 2.2 当前脚本能力边界

1. `install.sh` 负责安装 Docker、拉取镜像、启动容器
2. `uninstall.sh` 负责停止容器、删除数据，并支持保留数据或彻底清理
3. 当前 `install/README.md` 已明确安装和卸载，但没有形成同等级的升级与兼容执行规范
4. 因此，当前平台生命周期文档是“安装/卸载清楚，升级/迁移/兼容不足”

### 2.3 与旧架构的兼容矛盾

根据 [docs/current-architecture-baseline.md](../current-architecture-baseline.md)：

1. 更早期体系依赖宿主机 root、systemd、Cockpit 和多容器假设
2. 新安装脚本已经转向单容器运行时
3. 但升级兼容不能直接沿用旧假设，也不能默认旧安装会自动无缝迁移到新模型
4. 因此，平台安装与升级问题必须单独成文，不能只作为应用商店升级的附属说明

## 3. 为什么这部分要独立成包

这部分必须独立，原因很直接：

1. 安装、升级、卸载本质上属于平台生命周期治理，而不是发布治理本身
2. 它直接决定旧版到新版升级时，数据、配置和入口是否还能继续工作
3. 如果把这部分混进 DevOps 包，边界会再次混乱
4. 对用户而言，真正的高风险往往发生在升级迁移和卸载动作，而不是制品发布动作

## 4. 不可破坏约束

### 4.1 安装约束

1. 安装入口必须保持简单、单命令可执行
2. 安装参数必须有明确默认值和可追踪行为
3. 新增能力不能破坏已有 `install.sh` 用法

### 4.2 升级约束

1. 升级不能默认等同于全新安装
2. 升级必须明确保留哪些数据、配置、证书和入口
3. 升级失败必须有回退或至少有可恢复说明

### 4.3 卸载约束

1. 卸载必须明确区分“仅移除容器”和“彻底清理数据”
2. 卸载参数语义不能含糊
3. 卸载动作必须避免误删不属于当前产品的数据

### 4.4 兼容约束

1. 旧部署模型到新部署模型的迁移不能被默认假设为自动成功
2. 旧入口 URL、端口、证书、数据卷和用户数据必须在文档中明确处理策略
3. 升级兼容必须单独验证，不能只靠安装成功来代替

### 4.5 数据迁移保障约束

1. 旧版升级到新版后，必须先定义哪些数据需要迁移，不能在升级时临时判断
2. 数据迁移必须优先覆盖配置文件、数据卷、证书、入口参数和用户相关数据
3. 迁移过程必须区分“原地复用”“结构转换”“人工确认”三种类型
4. 未完成迁移验证前，不允许把升级路径标记为稳定可用

## 5. 本轮明确需求

这一专题当前应明确纳入以下需求：

1. 安装入口规范
2. 升级入口与升级前检查规范
3. 卸载模式与风险边界规范
4. 配置、数据卷、证书、入口 URL 的兼容策略
5. 从旧架构假设迁移到当前单容器模型的兼容策略
6. 旧版升级到新版后的数据迁移保障策略

当前不应混入：

1. 应用商店 catalog / library 数据制品规范
2. 应用安装业务流程本身
3. 与平台生命周期无关的前端交互优化

## 6. 推荐的执行边界

### 6.1 应写进本专题的内容

1. `install.sh`、`install_docker.sh`、`uninstall.sh` 的职责边界
2. 升级动作的最小契约
3. 卸载动作的最小契约
4. 兼容迁移与回退要求
5. 验证清单

### 6.2 不应写进本专题的内容

1. 应用商店数据版本与 manifest 细节

## 7. 第一阶段应交付什么

第一阶段不要求一次性解决全部迁移问题，但至少要交付：

1. 安装、升级、卸载三个入口的角色定义
2. 升级前检查项
3. 卸载风险分级
4. 配置、数据卷、证书、入口 URL 的兼容策略表
5. 数据迁移分类表
6. 最小验证清单

## 8. 升级后的数据迁移保障

升级从旧版到新版时，至少要显式覆盖以下五类数据：

| 数据类别 | 典型内容 | 默认策略 |
|---|---|---|
| 配置 | 安装参数、环境变量、INI/配置文件 | 优先原地复用，必要时做格式转换 |
| 数据卷 | 应用数据卷、平台持久卷 | 默认保留，不允许升级过程隐式删除 |
| 证书与入口 | 证书文件、域名、端口、入口 URL | 保持兼容优先，变更必须可回退 |
| 身份与凭据 | API key、用户配置、外部服务凭据 | 迁移前备份，迁移后校验 |
| 集成状态 | 代理、仓库、外部服务连接信息 | 升级后必须重新校验有效性 |

数据迁移策略至少分为三类：

1. 原地复用：结构不变，只做存在性和权限校验
2. 结构转换：旧字段或旧目录需要映射到新结构
3. 人工确认：存在高风险覆盖、不可逆变更或外部依赖时，禁止静默升级

升级设计至少要保证：

1. 迁移前有备份或快照点
2. 迁移中有步骤级日志
3. 迁移后能校验核心数据是否仍可用
4. 失败后有回退说明或恢复路径

## 9. 最小验证清单

至少应验证：

1. 全新安装成功
2. 升级不会破坏既有数据目录
3. 升级不会破坏既有入口配置
4. 卸载 `--keep-data` 行为符合预期
5. 卸载彻底清理行为符合预期
6. 旧模型迁移时有明确人工或半自动操作说明

## 10. 当前结论

平台本身的安装、升级、卸载和兼容问题，应该作为独立文档包处理，不再混入 `docs/standards/devops`。

原因不是它比发布治理更重要，而是它解决的是另一层问题：平台生命周期治理与升级迁移保障。
