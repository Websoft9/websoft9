## 1. 这个目录现在是干什么的

这个目录现在不是“照着脚本改”的操作说明，而是平台生命周期的重规划入口。

这里统一回答 5 个问题：

1. 旧系统到底是什么
2. 新系统到底是什么
3. 新版本的安装、升级、卸载应该遵守什么规范
4. 旧版到新版的一键升级应该如何定义
5. 哪些规则在文档审完之前不能进入实现

先看主文档：

1. [platform-lifecycle-governance_cn.md](./platform-lifecycle-governance_cn.md)

## 2. 当前判断

当前仓库里的安装脚本只能作为“现状样本”，不能作为未来规范的锚点。

原因很简单：

1. 旧系统是 Cockpit、多容器、systemd、宿主机控制模型
2. 新系统是单容器、Compose、容器内控制模型
3. 两代系统的控制面、运行面、数据面和入口面都发生了变化
4. 因此，新的安装、升级、卸载规范必须从系统事实和目标能力出发重写，不能沿着当前脚本小修小补

## 3. 这轮文档重规划要产出什么

本轮文档至少要把下面内容一次性理顺：

1. 旧系统与新系统的事实基线
2. 生命周期设计原则和行业最佳实践
3. 安装规范
4. 升级规范
5. 卸载规范
6. 资产迁移矩阵
7. 自动识别规则
8. 回退模型
9. 实施前审查门槛

当前状态说明：

1. 上述 9 项骨架已经补齐
2. 文档已经可以作为 install、upgrade、uninstall 脚本重写的设计输入
3. 后续实现仍必须遵守主文档里的混合环境限制和资产映射约束
