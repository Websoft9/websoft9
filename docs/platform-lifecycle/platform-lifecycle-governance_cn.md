# 平台安装升级迁移卸载说明

创建时间：2026-06-08
状态：可实施
适用对象：安装脚本维护者、发布负责人、平台运维
所属阶段：平台生命周期主文档

## 1. 先说结论

这份文档只讲 4 件事：

1. 怎么安装
2. 怎么升级
3. 升级时数据怎么迁移
4. 怎么卸载

如果你要看制品、版本、发布、回滚，去 [../standards/devops/README.md](../standards/devops/README.md)。

如果你要看应用商店本身属于平台什么能力，去 [../platform/app-store-foundation_cn.md](../platform/app-store-foundation_cn.md)。

## 2. 当前事实

以下内容以当前仓库可验证事实为基线。

### 2.1 当前安装入口

根据 [install/README.md](../../install/README.md)：

1. 当前对外安装入口是 `install/install.sh`
2. 当前支持独立 Docker 安装入口 `install/install_docker.sh`
3. 当前安装模型已经面向单容器运行时，而不是旧的 Cockpit 多容器部署模式

### 2.2 当前卸载入口

1. 当前对外卸载入口是 `install/uninstall.sh`
2. `uninstall.sh` 支持保留数据或彻底清理

### 2.3 当前升级短板

1. `install.sh` 负责安装 Docker、拉取镜像、启动容器
2. `uninstall.sh` 负责停止容器、删除数据，并支持保留数据或彻底清理
3. 当前 `install/README.md` 已明确安装和卸载，但没有形成同等级的升级与兼容执行规范
4. 因此，当前平台生命周期文档是“安装/卸载清楚，升级/迁移/兼容不足”

### 2.4 当前升级兼容风险

根据 [docs/current-architecture-baseline.md](../current-architecture-baseline.md)：

1. 更早期体系依赖宿主机 root、systemd、Cockpit 和多容器假设
2. 新安装脚本已经转向单容器运行时
3. 但升级兼容不能直接沿用旧假设，也不能默认旧安装会自动无缝迁移到新模型
4. 因此，平台安装与升级问题必须单独成文，不能只作为应用商店升级的附属说明

## 3. 安装怎么处理

1. 安装入口必须保持简单、单命令可执行
2. 安装参数必须有明确默认值和可追踪行为
3. 新增能力不能破坏已有 `install.sh` 用法

## 4. 升级怎么处理

1. 升级不能默认等同于全新安装
2. 升级必须明确保留哪些数据、配置、证书和入口
3. 升级失败必须有回退或至少有可恢复说明

升级前至少要检查：

1. 当前版本和目标版本
2. 配置文件是否需要迁移
3. 数据卷是否需要保留
4. 证书和入口参数是否会变化
5. 是否具备备份或回退点

### 4.1 升级前检查表

| 检查项 | 必须回答的问题 | 结果记录 |
|---|---|---|
| 当前版本 | 当前正在运行哪个版本 | 必填 |
| 目标版本 | 这次准备升级到哪个版本 | 必填 |
| 容器与卷状态 | 当前容器和卷是否健康、是否可备份 | 必填 |
| 配置文件 | 是否存在结构变更、是否需要转换 | 必填 |
| 证书与入口 | 域名、端口、证书是否会变化 | 必填 |
| 回退点 | 是否已完成快照、备份或旧镜像保留 | 必填 |
| 停机窗口 | 是否需要维护窗口与人工确认 | 视情况 |

### 4.2 升级实施步骤

第一批升级实施时，统一按下面顺序执行：

1. 记录当前版本、当前镜像和当前卷信息
2. 执行升级前检查表
3. 备份关键配置、卷、证书和入口参数
4. 按目标版本执行升级动作，不得把升级当成全新安装覆盖
5. 执行数据迁移动作
6. 验证入口、核心功能和数据可用性
7. 失败时按回退步骤恢复

## 5. 数据迁移怎么处理

升级从旧版到新版时，至少要显式覆盖以下五类数据：

| 数据类别 | 典型内容 | 默认策略 |
|---|---|---|
| 配置 | 安装参数、环境变量、INI/配置文件 | 优先原地复用，必要时做格式转换 |
| 数据卷 | 应用数据卷、平台持久卷 | 默认保留，不允许升级过程隐式删除 |
| 证书与入口 | 证书文件、域名、端口、入口 URL | 保持兼容优先，变更必须可回退 |
| 身份与凭据 | API key、用户配置、外部服务凭据 | 迁移前备份，迁移后校验 |
| 集成状态 | 代理、仓库、外部服务连接信息 | 升级后必须重新校验有效性 |

数据迁移只分三类，别再拆得更细：

1. 原地复用：结构不变，只做存在性和权限校验
2. 结构转换：旧字段或旧目录需要映射到新结构
3. 人工确认：存在高风险覆盖、不可逆变更或外部依赖时，禁止静默升级

升级设计至少要保证：

1. 迁移前有备份或快照点
2. 迁移中有步骤级日志
3. 迁移后能校验核心数据是否仍可用
4. 失败后有回退说明或恢复路径

### 5.1 数据迁移矩阵

| 数据类别 | 典型位置 | 默认动作 | 是否允许自动执行 |
|---|---|---|---|
| 安装参数与环境变量 | 安装参数、env、INI | 原地复用或结构转换 | 是 |
| 平台持久卷 | Docker volume、挂载目录 | 保留并校验 | 是 |
| 应用数据卷 | 应用运行数据 | 保留并校验 | 是 |
| 证书文件 | SSL 证书、私钥、域名映射 | 保留并校验 | 是 |
| 入口配置 | 端口、URL、反向代理入口 | 保留优先，必要时转换 | 是 |
| 外部凭据 | API key、第三方接入信息 | 备份后校验 | 视情况 |
| 高风险自定义改动 | 人工修改脚本、宿主机特殊依赖 | 禁止静默覆盖 | 否 |

### 5.2 迁移后验证

迁移完成后，至少验证：

1. 入口 URL 仍可访问
2. 容器与核心服务能启动
3. 原有卷中的关键数据仍存在
4. 证书与端口没有被错误覆盖
5. 外部集成仍能连接

## 6. 卸载怎么处理

1. 卸载必须明确区分“仅移除容器”和“彻底清理数据”
2. 卸载参数语义不能含糊
3. 卸载动作必须避免误删不属于当前产品的数据

### 6.1 卸载模式说明

| 模式 | 入口 | 结果 |
|---|---|---|
| 标准卸载 | `sudo bash uninstall.sh` | 移除容器与默认数据 |
| 保留数据卸载 | `sudo bash uninstall.sh --keep-data` | 移除容器，保留卷与数据 |
| 彻底清理 | `sudo bash uninstall.sh --purge` | 移除容器、数据与镜像 |

## 7. 回退怎么处理

升级失败时，统一按下面顺序回退：

1. 停止本次升级后异常的容器或服务
2. 恢复升级前保留的配置、卷或快照
3. 恢复上一个稳定版本的镜像或运行参数
4. 验证入口、证书、核心服务与关键数据
5. 记录失败原因，不得在未定位原因前重复覆盖升级

## 8. 第一批实施文件落点

第一批实施统一只改下面这些文件：

| 目的 | 文件 |
|---|---|
| 安装主入口 | `install/install.sh` |
| Docker 安装入口 | `install/install_docker.sh` |
| 卸载入口 | `install/uninstall.sh` |
| 当前安装说明 | `install/README.md` |
| 发布附带安装脚本 | `.github/workflows/release.yml` |

说明：

1. 升级前检查、迁移矩阵和回退步骤先写进本文
2. 真正改脚本时，只围绕上面这几个入口，不额外扩面
3. 如果涉及运行容器内部行为，再单独记录受影响的运行时脚本

## 9. 最少要交付什么

这一块不用一次写成很多小文档，最少把下面内容写清楚就够：

1. 安装、升级、卸载三个入口的角色定义
2. 升级前检查项
3. 卸载风险分级
4. 配置、数据卷、证书、入口 URL 的兼容策略表
5. 数据迁移分类表
6. 最小验证清单

## 10. 最小验证清单

至少应验证：

1. 全新安装成功
2. 升级不会破坏既有数据目录
3. 升级不会破坏既有入口配置
4. 卸载 `--keep-data` 行为符合预期
5. 卸载彻底清理行为符合预期
6. 旧模型迁移时有明确人工或半自动操作说明

## 11. 当前结论

平台这一块不要再拆很多小文档，保持“1 个入口 + 1 份主文档”就够。

这份主文档只需要让人一眼看明白四件事：怎么安装、怎么升级、数据怎么迁移、怎么卸载。
