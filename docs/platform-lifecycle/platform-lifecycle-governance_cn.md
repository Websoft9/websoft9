# 平台安装升级卸载重规划

创建时间：2026-06-08
最近重写：2026-06-11
状态：已具备进入实施设计条件
适用对象：产品负责人、架构负责人、安装脚本维护者、发布负责人、平台运维
所属阶段：平台生命周期主文档

## 1. 文档目标

这份文档不是对现有安装脚本的解释，而是 Websoft9 平台生命周期的重新规划基线。

它只回答 4 类问题：

1. 新版本安装应该怎么规范
2. 新版本之间的迭代升级应该怎么规范
3. 旧版本升级到当前新版本应该怎么规范
4. 卸载与回退应该怎么规范

这份文档的约束前提只有一条：

1. 当前仓库里的 install 脚本只能作为现状参考，不能作为未来规范的锚点

如果你要看制品、版本、发布、回滚，去 [../standards/devops/README.md](../standards/devops/README.md)。

如果你要看应用商店本身属于平台什么能力，去 [../platform/app-store-foundation_cn.md](../platform/app-store-foundation_cn.md)。

## 2. 重规划为什么成立

这次不是在现有安装脚本上修修补补，而是允许重新规划，原因有 3 个：

1. 旧系统和新系统不是同一安装模型，不能沿着旧逻辑平移
2. 当前新脚本只覆盖了当前单容器模型的最窄场景，不能自然外推成长期规范
3. 未来还要同时覆盖“旧版升级到新版”和“新版持续迭代升级”两条路径，所以必须先定义统一生命周期模型

## 3. 两代系统的事实基线

### 3.1 旧系统事实

基于 main 分支和旧文档、旧脚本可验证事实，旧系统具有下面特征：

1. 控制面以 `install.sh`、`install_cockpit.sh`、systemd、Cockpit 为核心
2. UI 以 Cockpit 插件为核心，而不是产品自带单一前端壳
3. 运行面是多容器：至少包含 `websoft9-apphub`、`websoft9-deployment`、`websoft9-git`、`websoft9-proxy`
4. 宿主机依赖较重：依赖 Cockpit、systemd、宿主机命令桥接和宿主机配置
5. 数据面分散在 AppHub、Gitea、Portainer、Nginx Proxy Manager 等多个卷与配置目录中
6. 卸载不仅涉及容器，还涉及 systemd、Cockpit 包、Cockpit 插件目录和宿主机文件

### 3.2 新系统事实

基于当前仓库可验证事实，新系统具有下面特征：

1. 对外安装入口收敛到 `install/install.sh`
2. Docker 安装能力收敛到 `install/install_docker.sh`
3. 卸载入口收敛到 `install/uninstall.sh`
4. 运行面收敛到 `docker/docker-compose.yml` 中的 `websoft9` 单容器
5. 持久化模型收敛到 `product_data` 单卷；历史 `product_custom`、`product_logs`、`product_letsencrypt` 与 `product_modsec` 仅保留升级回滚兼容识别，不再作为当前运行时挂载
6. 当前升级能力只覆盖当前单容器版本之间的升级，不覆盖旧系统到新系统

### 3.3 旧新系统差异为什么重要

旧系统和新系统的变化不是“部署从多容器变单容器”这么简单，而是 4 个层面同时变化：

1. 控制面变了：Cockpit + systemd 变成 Compose + 单入口脚本
2. 运行面变了：多容器变成单容器
3. 数据面变了：多类卷与多服务数据需要重新映射到新持久化模型
4. 入口面变了：Cockpit 登录入口与旧代理入口不再等于当前产品入口

因此，新的安装、升级、卸载规范必须先定义统一模型，再决定脚本如何实现。

## 4. 规划原则

新文档和未来脚本统一遵守下面 8 条原则。

### 4.1 产品原则

1. 用户入口必须尽量简单，安装和升级都要保持单入口命令体验
2. 文档必须区分“当前已具备能力”和“目标能力”，不能混写
3. 升级必须被视为一级能力，不是安装的附属分支

### 4.2 工程原则

1. 先定义资产模型，再定义迁移动作
2. 先定义回退边界，再定义自动执行边界
3. 先定义识别规则，再定义升级策略

### 4.3 结合行业最佳实践后的简化原则

结合 Docker 官方文档，可提炼出对 Websoft9 最实用的简化原则：

1. 持久数据优先使用命名卷，不把关键持久状态放在容器可写层
2. 卷的备份、恢复、迁移必须作为升级前置动作，而不是失败后的补救动作
3. 容器生命周期与数据生命周期要分离，删除容器不等于删除数据
4. 宿主机级 process manager 和 Docker 自身重启策略不要混用
5. 通道策略要明确，稳定版和预发布版的升级风险不能混淆
6. 不把复杂度扩散到用户命令行，复杂性应收敛到脚本内部和文档规则中

## 5. 生命周期统一模型

未来平台统一只定义 3 条主路径：

1. 全新安装
2. 升级
3. 卸载

其中“升级”必须再拆成两个一级子路径：

1. 旧版升级到当前新版
2. 当前新版内部的后续迭代升级

### 5.1 全新安装

全新安装的目标很简单：

1. 在受支持 Linux 上安装或校验 Docker
2. 准备标准化部署物料
3. 写入标准化配置
4. 启动单容器产品
5. 校验入口和健康状态

### 5.2 旧版到新版升级

这是一次跨安装模型升级，不是普通版本升级。

这条路径必须同时完成：

1. 自动识别旧环境
2. 自动建立备份点
3. 资产映射与迁移
4. 新入口接管
5. 校验与回退

### 5.3 新版迭代升级

这是同一安装模型内的升级。

它应该尽可能比旧版迁移简单，重点是：

1. 保留命名卷
2. 更新部署物料和镜像版本
3. 保持配置兼容
4. 做最小必要的数据结构迁移
5. 保留可回退点

### 5.4 卸载

卸载必须和升级模型解耦，只回答 3 件事：

1. 移除运行实体
2. 是否保留持久数据
3. 是否清理历史制品和镜像

## 6. 版本识别与环境识别规范

新旧版本识别不能依赖文档说明，必须基于运行时和宿主机事实。

### 6.1 识别输入信号

第一版识别规则至少包含 5 类信号：

| 信号类别 | 旧系统典型信号 | 新系统典型信号 | 作用 |
|---|---|---|---|
| 容器信号 | `websoft9-apphub`、`websoft9-deployment`、`websoft9-git`、`websoft9-proxy` | `websoft9` | 判断运行拓扑 |
| systemd 信号 | `cockpit`、`cockpit.socket`、`websoft9` | 通常无强依赖 | 判断宿主机控制面 |
| 安装路径信号 | `/data/websoft9/source`、旧脚本目录、旧 compose 文件 | `/opt/websoft9` 或未来标准路径 | 判断部署模型 |
| 配置信号 | 旧 `.env`、cockpit 配置、旧代理配置、旧 INI | 新 `.env`、标准 compose、容器内配置 | 判断配置模型 |
| 卷与入口信号 | `gitea`、`portainer`、`nginx_*` 卷与旧端口 | `product_*` 卷与新入口端口 | 判断数据面和入口面 |

### 6.2 识别输出状态

识别结果只允许输出 4 种状态：

1. `legacy`：明确是旧系统
2. `modern`：明确是新系统
3. `mixed`：旧新资产并存
4. `unknown`：无法可靠判定

### 6.3 识别策略约束

1. 只有 `legacy` 才允许进入旧版到新版迁移路径
2. 只有 `modern` 才允许进入新版迭代升级路径
3. `mixed` 和 `unknown` 不能静默升级，必须转人工确认或专门治理流程

## 7. 安装规范

新的安装规范只保留最少必要能力。

### 7.1 安装目标

1. 单入口命令
2. 最少必要参数
3. 明确默认值
4. 清晰日志
5. 安装完成后可直接验证

### 7.2 安装前检查

安装前至少检查：

1. OS 是否受支持
2. Docker 是否存在或可安装
3. 关键端口是否可用或可重映射
4. 目标路径是否可写
5. 磁盘空间是否足够
6. 网络与制品源是否可达

### 7.3 安装边界

1. 不把旧版迁移逻辑塞进全新安装路径
2. 不把未来未实现的升级能力写成已支持能力
3. 不把用户自定义场景默认纳入第一版安装能力

### 7.4 安装实施检查表

在脚本进入实现前，安装路径至少要明确下面内容：

| 检查项 | 必须回答的问题 | 通过标准 |
|---|---|---|
| 受支持平台 | 哪些 Linux 发行版进入第一批支持范围 | 有明确白名单 |
| Docker 策略 | Docker 缺失时自动安装还是中止 | 有单一规则 |
| 端口策略 | 9000、80、443 冲突时如何处理 | 有明确分流 |
| 路径策略 | 安装目录、日志目录、部署物料目录是否固定 | 有固定默认值 |
| 通道策略 | stable、rc、dev 是否有清晰默认值 | 有清晰通道规则 |
| 成功判定 | 安装成功依赖哪些最小检查 | 有统一健康标准 |

## 8. 升级规范

升级是整个生命周期里最重要的部分，必须单独设计。

### 8.1 升级分类

| 升级类型 | 来源 | 目标 | 性质 |
|---|---|---|---|
| 迁移升级 | 旧系统 | 当前新系统 | 跨模型升级 |
| 迭代升级 | 当前新系统 | 后续新系统版本 | 同模型升级 |

### 8.2 两类升级的共同要求

两类升级都必须具备下面能力：

1. 升级前识别
2. 升级前备份
3. 变更日志
4. 升级后校验
5. 失败回退

### 8.3 旧版到新版一键升级的定义

旧版到新版的一键升级，不是指“脚本只有一段命令”，而是指用户视角上只执行一个升级入口，而脚本内部自动完成下列动作：

1. 识别旧系统
2. 记录旧环境状态
3. 建立配置、卷、证书和部署物料的备份点
4. 按资产映射规则迁移数据
5. 启动新系统
6. 校验核心能力
7. 失败时自动或半自动回退

### 8.4 新版迭代升级的定义

新版迭代升级应被设计成比迁移升级更窄、更稳：

1. 优先兼容已有配置
2. 优先复用已有命名卷
3. 尽量避免要求用户重新输入安装参数
4. 数据结构迁移必须有版本感知和回退策略

### 8.5 升级非目标

第一版规划中，下面内容不应被当作默认目标：

1. 自动吸收所有宿主机级个性化脚本
2. 自动兼容所有未知或混合状态环境
3. 在无备份条件下继续强制升级

### 8.6 跨代升级检查表

旧系统到新系统的一键升级，在实施前至少要形成下面检查表：

| 检查项 | 需要确认的事实 | 结果要求 |
|---|---|---|
| 环境识别 | 是否明确识别为 `legacy` | 必须 |
| 旧入口 | Cockpit 端口、80/443 是否已记录 | 必须 |
| 旧控制面 | Cockpit、systemd、旧插件目录是否存在 | 必须 |
| 旧卷 | `apphub_logs`、`apphub_config`、`portainer`、`gitea`、`nginx_*` 是否已清单化 | 必须 |
| 旧配置 | 旧 `.env`、INI、宿主机配置是否已备份 | 必须 |
| 证书 | 旧证书目录和代理证书是否已备份 | 必须 |
| 回退点 | 是否已保留旧运行时恢复条件 | 必须 |
| 切换后校验 | 入口、关键数据、关键服务是否有统一验证项 | 必须 |

### 8.7 迭代升级检查表

新系统版本之间升级，在实施前至少要形成下面检查表：

| 检查项 | 需要确认的事实 | 结果要求 |
|---|---|---|
| 环境识别 | 是否明确识别为 `modern` | 必须 |
| 版本差异 | 镜像、部署物料、配置 schema 是否有差异 | 必须 |
| 卷兼容 | `product_*` 卷是否可直接复用 | 必须 |
| 参数兼容 | 现有 `.env` 是否可原地升级 | 必须 |
| 数据迁移 | 是否存在容器内结构升级脚本 | 视版本 |
| 回退点 | 升级前镜像、配置、部署物料是否已保留 | 必须 |
| 切换后校验 | 健康检查、入口可达、核心数据存在性是否通过 | 必须 |

## 9. 资产模型与迁移矩阵

升级设计必须先有资产模型，再有动作设计。

### 9.1 资产分类

统一只定义 6 类资产：

1. 入口资产
2. 配置资产
3. 数据资产
4. 证书资产
5. 身份与凭据资产
6. 运维痕迹资产

### 9.1A 能力来源约束

资产模型之外，还必须同时看能力来源模型。

基于 [install/current-product-architecture-baseline.md](../../install/current-product-architecture-baseline.md) 与 [install/legacy-main-architecture-baseline.md](../../install/legacy-main-architecture-baseline.md) 的代码事实基线，当前至少要区分 3 类能力来源：

1. 原生产品能力：主要由 Websoft9 自己的前端、后端和运行时逻辑实现
2. 集成封装能力：由 Websoft9 提供页面与业务编排，但执行依赖 Portainer、Gitea、Nginx Proxy Manager 等第三方系统
3. 第三方入口能力：Websoft9 主要提供统一入口、壳或会话桥接，实际功能由第三方产品本身提供

这条约束会直接影响迁移设计：

1. 原生产品能力优先看产品配置、身份、日志和内部状态兼容
2. 集成封装能力必须成对检查“产品侧元数据”和“第三方系统状态”
3. 第三方入口能力默认优先处理入口替换和数据存续，不应误写成复杂的产品原生数据迁移

### 9.2 第一版迁移矩阵

| 资产类别 | 旧系统典型来源 | 新系统目标位置 | 默认动作 | 风险级别 |
|---|---|---|---|---|
| 入口资产 | Cockpit 端口、80/443 代理入口 | 新版 9000、80、443 入口模型 | 保留优先，必要时转换 | 高 |
| 配置资产 | 旧 `.env`、INI、宿主机配置 | 新 `.env`、compose、容器内配置 | 映射转换 | 高 |
| 数据资产 | AppHub、Gitea、Portainer、NPM 卷 | `product_*` 或未来标准卷 | 保留并迁移 | 高 |
| 证书资产 | NPM 证书目录、代理证书文件 | 新版证书卷 | 备份后导入 | 高 |
| 身份与凭据资产 | 旧 API key、服务凭据、登录信息 | 新版凭据模型 | 备份并校验 | 中 |
| 运维痕迹资产 | systemd、Cockpit 包、插件目录、宿主机脚本 | 新版不再继承的资产 | 清理或保留说明 | 中 |

### 9.3 自动化边界

只有满足下面条件的资产才允许自动迁移：

1. 来源明确
2. 目标明确
3. 转换规则明确
4. 校验规则明确
5. 回退方式明确

缺少任一项，都不应进入自动迁移清单。

### 9.4 第一版正式资产映射表

下面这张表是进入实施前必须冻结的第一版资产映射基线。

| 资产类别 | 旧系统来源 | 新系统目标 | 自动动作 | 校验方式 | 回退方式 |
|---|---|---|---|---|---|
| 旧控制台入口 | Cockpit `ListenStream` 与旧 `CONSOLE_PORT` | 新 `CONSOLE_PORT` 对应 9000 入口 | 识别旧端口并映射到新入口策略 | 新入口可达，旧端口占用状态可解释 | 恢复旧 Cockpit 监听和旧代理入口 |
| HTTP/HTTPS 入口 | 旧 `APP_HTTP_GATEWAY`、`APP_HTTPS_GATEWAY` 与 `websoft9-proxy` 暴露端口 | 新 80/443 入口 | 复用或转换端口映射 | 80/443 可达，代理链路正常 | 恢复旧 proxy 暴露规则 |
| AppHub 配置 | `apphub_config` 卷挂载到 `/websoft9/apphub/src/config` | 新 `.env` 与容器内运行时配置模型 | 提取并转换必要配置键 | 新容器启动成功，配置项生效 | 恢复旧 `apphub_config` 备份并启动旧运行时 |
| AppHub 日志 | `apphub_logs` 卷挂载到 `/websoft9/apphub/logs` | `product_data:/data` 下的 `/data/logs/legacy-apphub` 子树 | 归档或合并日志 | 日志可读取，归档不丢失 | 恢复旧日志卷挂载 |
| Gitea 数据 | `gitea:/data` | `product_data:/data` 下的 `/data/gitea` 子树 | 复制或导入到新数据根 | 关键目录与元数据存在 | 恢复旧 `gitea` 卷并切回旧运行时 |
| Portainer 数据 | `portainer:/data` | `product_data:/data` 下的 `/data/portainer` 子树 | 复制或导入到新数据根 | 目标目录存在且权限正确 | 恢复旧 `portainer` 卷 |
| 代理业务数据 | `nginx_data:/data` | `product_data:/data` 下的 `/data/nginx-proxy-manager` 或 `/data/nginx/*` | 分目录迁移 | 代理配置与主机条目存在 | 恢复旧 `nginx_data` 卷 |
| 证书 | `nginx_letsencrypt:/etc/letsencrypt` | 升级期兼容备份资产，不再挂载到现代运行时 | 仅在升级回滚或人工提取时保留归档 | 证书归档可读取 | 恢复旧 `nginx_letsencrypt` 卷 |
| ModSecurity 配置 | `nginx_modsec:/etc/modsec` | 升级期兼容备份资产，不再挂载到现代运行时 | 仅在升级回滚或人工提取时保留归档 | 归档可读取 | 恢复旧 `nginx_modsec` 卷 |
| 自定义代理配置 | `nginx_var:/etc/custom` | `product_data:/data` 下的 `/data/config` 子树 | 复制到统一数据根并保留兼容软链接 | 自定义配置存在，入口正常 | 恢复旧 `nginx_var` 卷 |
| 旧部署物料 | 旧 compose、旧 `.env`、旧安装目录 `/data/websoft9/source` | 新安装目录和标准部署物料 | 归档旧物料并生成新物料 | 新物料完整，旧物料已归档 | 恢复旧 compose 和旧 `.env` |
| 旧控制面遗留 | `/etc/cockpit`、`/usr/share/cockpit`、`/lib/systemd/system/websoft9.service`、Cockpit 包 | 新系统不再继承 | 切换成功后按策略停用或清理 | 新系统独立运行，旧控制面状态已记录 | 重新启用旧 systemd 与 Cockpit |

实施约束：

1. 上表中任一行如果“自动动作、校验方式、回退方式”不明确，就不能进入自动化实现
2. 第一版实现允许把部分行降级为“半自动”，但不能把未定义行静默跳过

## 10. 卸载规范

卸载规范必须同时照顾当前新系统和未来长期维护。

### 10.1 卸载模式

统一定义 3 种模式：

| 模式 | 目标 | 数据处理 |
|---|---|---|
| 停用 | 停止运行但保留恢复条件 | 保留数据 |
| 标准卸载 | 移除运行实体与当前部署物料 | 保留或按选项处理数据 |
| 彻底清理 | 移除运行实体、部署物料、镜像和明确归属的数据 | 删除明确归属数据 |

### 10.2 卸载原则

1. 卸载不能误删不属于 Websoft9 的宿主机资产
2. 数据删除必须显式选择，不能靠默认副作用发生
3. 卸载结果必须可解释，让用户知道哪些被删、哪些被保留

### 10.3 卸载分级检查表

| 卸载级别 | 必须删除 | 默认保留 | 需人工确认 |
|---|---|---|---|
| 停用 | 当前运行容器或服务 | 卷、证书、配置、镜像 | 无 |
| 标准卸载 | 运行容器、当前部署物料 | 可保留卷、证书、日志、镜像 | 删除数据前必须确认 |
| 彻底清理 | 运行容器、部署物料、卷、镜像、平台标识 | 无 | 遇到混合环境时必须确认 |

### 10.4 旧系统与混合环境卸载约束

1. 如果识别为 `legacy` 或 `mixed`，卸载不能直接复用新系统清理逻辑
2. 是否移除 Cockpit 包、旧 systemd、旧插件目录，必须作为独立步骤设计
3. 任何涉及旧入口和旧证书删除的步骤，都必须在回退窗口关闭后再执行

## 11. 回退模型

任何升级规划都必须先设计回退。

### 11.1 最小回退单元

文档上需要明确 4 层回退单元：

1. 配置回退
2. 卷与数据回退
3. 部署物料和镜像回退
4. 入口切换回退

### 11.2 回退原则

1. 升级前必须先准备回退点
2. 回退路径必须和升级路径一起设计，不允许后补
3. 回退成功的定义必须和升级成功一样可验证

### 11.3 回退单元说明表

| 回退单元 | 典型内容 | 触发条件 | 回退动作 |
|---|---|---|---|
| 配置回退 | `.env`、运行参数、配置文件转换结果 | 新容器启动失败或配置错误 | 恢复升级前配置并重启旧/新运行时 |
| 入口回退 | 9000、80、443、旧 Cockpit 端口与代理绑定 | 切换后入口不可达 | 恢复旧端口绑定和旧代理入口 |
| 数据卷回退 | `product_*` 与旧卷迁移结果 | 关键数据缺失、权限异常、结构错误 | 恢复备份卷或切回旧卷 |
| 运行时回退 | compose、镜像标签、容器编排状态 | 升级后健康检查失败 | 恢复旧 compose、旧镜像、旧容器 |

### 11.4 跨代升级额外回退要求

1. 旧系统运行时在新系统校验完成前，不应被不可逆删除
2. 旧 Cockpit/systemd 退场动作必须放在“新系统稳定”之后，而不是切换之前
3. 旧版到新版的一键升级如果无法做到自动回退，至少要做到半自动一键恢复旧运行时

### 11.5 生命周期路径回退口径

为了避免后续脚本实现时重新发明口径，四条路径的回退边界统一定义如下：

1. 全新安装：只面向 `empty` 环境，失败后只清理本次安装生成的运行实体和临时物料，不承担旧资产恢复
2. 新版迭代升级：失败后优先恢复升级前 `.env`、compose、镜像和主数据卷备份点
3. 旧版到新版迁移升级：在新系统完成入口、关键数据和关键服务校验前，不得不可逆删除旧控制面、旧卷和旧部署物料
4. 卸载：不属于回退路径，但任何涉及旧 Cockpit、旧 systemd 和旧证书删除的动作，都必须在用户明确确认下执行

## 15. 当前结论

当前阶段最重要的结论只有 3 条：

1. 现有 install 脚本不应主导新规范，只能作为现状参考
2. 新规范必须同时覆盖旧版迁移升级和新版迭代升级
3. 以当前文档状态，已经可以进入实施设计，但实现必须严格受本文的识别、映射、校验和回退约束控制
