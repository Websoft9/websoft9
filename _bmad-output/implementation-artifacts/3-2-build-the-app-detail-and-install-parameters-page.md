# Story 3.2: Build the app detail and install-parameters page

## Status

done

## Story

As a user preparing to install an application,
I want to inspect app details and install parameters first,
so that installation decisions are informed rather than blind.

## Acceptance Criteria

1. Given the user opens an app detail page from the App Store, when the page loads, then the UI shows description, template or image information, required parameters, and related links, and the detail page stays clearly separated from submission actions.
2. Given some parameters come from media or catalog metadata, when the detail page renders the install form model, then the backend contract provides those definitions, and the frontend does not hardcode installation parameter shapes.

## Dependencies

- Story 3.1 already establishes the native App Store route, catalog browsing surface, and app payload loading baseline.
- The App Store runtime asset flow now provides `/media/json/product_{locale}.json`, `/media/json/catalog_{locale}.json`, and `/media/json/app-store-install-metadata.json` as the primary browse and install-metadata contract.
- AppHub still owns dynamic actions such as `/api/apps/install` and `/api/settings/domain`.

## Previous Story Intelligence

- The user explicitly steered the new App Store back toward the legacy plugin interaction model rather than a redesigned shell-native card/detail flow.
- The final Story 3.1 implementation already absorbed most Story 3.2 scope by merging detail and install into one modal sequence while keeping browsing, inspection, and submission as distinct steps.
- Install parameter shapes are no longer inferred from hardcoded frontend fields alone: runtime-generated metadata now carries `settings`, `is_web_app`, and `initial_apps` filtering into the first-party shell.

## Developer Context

- The controlling frontend surface is `console/src/features/app-store/app-store-page.tsx`, where the selected app now opens a detail-first modal and then transitions into install mode only after explicit user intent.
- The frontend parameter model is driven from `AppStoreApp.settings`, `AppStoreApp.is_web_app`, and distribution metadata supplied by `console/src/features/app-store/use-app-store-apps.ts` plus `console/src/features/app-store/app-store-model.ts`.
- Runtime install metadata is generated by `docker/product/scripts/platform-sync-runtime-assets.py`, which scans library `.env` files and emits `/media/json/app-store-install-metadata.json`.
- Product-owned domain behavior for web applications is resolved through `/api/settings/domain`, which allows the install flow to reuse current runtime configuration instead of browser-derived guesses.

## Implementation Guardrails

- Keep the legacy user mental model: inspect first, submit second.
- Do not collapse detail rendering and install submission into one blind action.
- Do not reintroduce hardcoded install parameter inventories for individual apps.
- Do not move install parameter derivation back into per-request AppHub browse transformation as the primary path.

## Dev Notes

### Architecture Compliance

- The detail/install experience remains inside the first-party React App Store route rather than falling back to legacy Cockpit runtime behavior.
- Runtime-generated JSON metadata is now the primary install-form definition source, which keeps the frontend generic while reducing AppHub browse-time transformation load.
- Dynamic submission still stays on the AppHub install endpoint; Story 3.2 only covers inspection and parameter preparation.

### API / Data Contract Notes

- Install parameter definitions are currently sourced from `/media/json/app-store-install-metadata.json`, generated from library `.env` files.
- Detail content uses the merged app payload assembled from static product JSON plus install metadata.
- Product-owned domain behavior depends on `/api/settings/domain` being readable by the first-party shell.
